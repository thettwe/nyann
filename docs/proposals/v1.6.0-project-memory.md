# v1.6.0 — Project Memory

**Status:** Implemented (shipped in v1.6.0)
**Owner:** project maintainer
**Last updated:** 2026-05-06

This proposal sits on top of [`docs/principles/documentation.md`](../principles/documentation.md),
which defines **Project Memory** — nyann's named documentation concept,
its five properties, and the dual-audience (AI-first, human-readable)
positioning. The proposal cites the principles doc as authority for
every design decision below; if a tradeoff seems arbitrary, it's
because the principles doc is doing the load-bearing work.

## Problem

nyann's documentation system has four overlapping issues, all visible
in production today.

### 1. Scaffolding is stack-driven, not codebase-aware

A Spring Boot API service and a Spring Boot CLI tool both bootstrap
with the `java-spring-boot` profile and get the same three templates:
`architecture`, `adrs`, `research`. The API service has no `runbook`,
no `api-reference`. The CLI has no command tutorials. The architecture
template prints "built on Spring Boot" but the rest of the docs use
zero codebase information.

Across all 18 starter profiles, scaffolding picks from three patterns:
`[architecture, adrs]`, `[architecture, adrs, research]`, or
`[architecture, prd, adrs, research]`. Selection is by *language and
framework*, never by *what kind of system this is*.

### 2. The template set is too narrow

Five templates total (`architecture`, `prd`, `adrs`, `research`,
`memory`). Common doc types absent across every profile:

- **API reference** — every API service needs an endpoint catalog.
  AI agents debugging endpoint behaviour have no scoped doc to retrieve.
- **Runbook** — anything in production needs an operational playbook.
  AI agents debugging an outage have nowhere to pull from.
- **Deployment** — currently smushed into architecture. Deserves its
  own scoped doc; AI agents asking "how does this ship?" need a
  bounded answer.
- **Glossary** — domain terms with definitions, invariants, and
  cross-references. The sleeper hit for AI second-brain (agents
  constantly mis-name domain concepts; one read fixes it).

### 3. The `memory/` folder is mis-framed

`templates/memory/README.tmpl` calls itself "Session-scratch for Claude.
What Claude should remember between turns or across sessions." That
framing collides with two things:

- **Claude Code's own auto-memory** at
  `~/.claude/projects/<encoded>/memory/` is what actually persists
  Claude's between-session memory. nyann's `memory/` is *team-shared*
  notes in the repo, a different layer.
- **Project Memory** (the new framing) lives in `docs/`, not
  `memory/`. The `memory/` folder is the *ephemeral scratch* layer
  beneath it.

Users reading the scaffolded README think nyann is duplicating
Claude's auto-memory. It isn't, but the framing doesn't make that
clear.

### 4. nyann fails its own doctor check

`doctor` flags this repo's `CLAUDE.md` as `warn` (4748 B vs 3072 B
soft cap, 8192 B hard cap). The "Architecture at a glance" (1823 B)
and "Non-negotiable conventions" (1277 B) sections are content, not
router. The principles doc says CLAUDE.md is router-mode. We're not
following our own rule.

### 5. Drift-check preamble duplicated 4×

The `## 0. Drift check` preamble in `commit/SKILL.md`,
`release/SKILL.md`, `pr/SKILL.md`, and `ship/SKILL.md` is identical
except for the verb (`commit flow` / `release flow` / `PR flow` /
`ship flow`). Future drift-check changes need 4 edits, with wording
drift inevitable.

## Goals (P0)

- **Ship Project Memory framing.** `docs/principles/documentation.md`
  becomes the foundation for all future doc-related decisions.
  README positioning section added so users understand what nyann's
  docs system actually is.
- **Codebase-aware scaffolding.** `StackDescriptor` gains an
  `archetype` field. Detection covers seven archetypes
  (`api-service`, `cli-tool`, `library`, `web-app`, `mobile-app`,
  `plugin`, `unknown`). Per-archetype scaffold maps drive which
  templates ship.
- **Four new templates** — `api-reference`, `runbook`, `deployment`,
  `glossary` — each scoped to specific archetypes, each aligned to
  Project Memory principles (bounded scope, predictable structure,
  codebase-aware variables).
- **Storage-agnostic.** New templates work identically for local,
  Obsidian, and Notion backends. Same Markdown source; per-doc-type
  routing already supported by `DocumentationPlan.targets[]`.
- **Backward compatible.** Archetype-aware scaffolds are opt-in via
  a new profile flag (`use_archetype_scaffolds: false` by default).
  Existing users see zero scaffold changes on v1.6.0 upgrade.
- **Memory framing fix.** Rewrite `templates/memory/README.tmpl` and
  the `gen-claudemd.sh` memory section to position `memory/` as the
  ephemeral scratch layer, not Claude's session memory.
- **CLAUDE.md self-compliance.** This repo's `CLAUDE.md` brought
  under the 3 KB soft cap. Extracted content moves to `docs/`.
- **Drift-check dedup.** `bin/session-check.sh` accepts a `--flow`
  argument and emits the formatted nudge. The 4 SKILL.md preambles
  trim to a 2-line invocation.

## Non-goals (explicit)

- **Eight new templates.** Deferred for v1.7+: `tutorials`,
  `command-reference`, `migration-guides`, `data-model`, `security`,
  `examples`. Better to ship four well-fitting templates than ten
  generic ones (per principles doc § *Adding a new doc template*).
- **`data-pipeline` archetype.** Detection signals are noisy
  (DVC vs Airflow vs Prefect vs Dagster all look different) and we
  have no users asking. Add when there's demand.
- **Default-on archetype scaffolds.** Opt-in for v1.6.0. Flipping
  the default to `true` is a v1.7+ decision once templates have
  soaked.
- **Archetype-specific drift checks** ("every endpoint must appear
  in `api-reference.md`", "every documented CLI command must match
  `--help`"). Future work.
- **Glossary auto-population from detected types.** Could nyann seed
  `glossary.md` from Go interfaces / TypeScript types / Python
  classes? Likely yes for v1.7+. Out of scope here.
- **Notion-block-formatted template variants.** Obsidian and Notion
  users get the same Markdown today. Native-format variants are a
  v1.7+ question.
- **Removing existing templates** (`architecture`, `prd`, `adrs`,
  `research`, `memory`). They predate the principles doc and are
  grandfathered in.
- **Auto-detecting archetype across multi-archetype monorepos**
  (e.g. a repo with one API service workspace and one CLI workspace).
  v1.6.0 emits a single archetype per StackDescriptor; multi-archetype
  is a v1.7+ design problem. For monorepos, profile authors can
  override via the `archetype` field.

## User flow

### Before (today)

```bash
$ /nyann:bootstrap
[detects: java + spring boot + maven]
[suggests: java-spring-boot profile]
[scaffolds: docs/architecture.md, docs/decisions/, docs/research/]
```

Three docs. None of them mention this is an API service. No endpoint
reference. No runbook. AI agents working on the codebase pull
`architecture.md` and learn the framework name; everything beyond
that they have to discover from code each session.

### After (proposed)

```bash
$ /nyann:bootstrap
[detects: java + spring boot + maven, archetype: api-service]
[suggests: java-spring-boot profile]
[asks: "Enable archetype-aware scaffolds (recommended)? [Y/n]"]
[scaffolds: architecture.md, api-reference.md, runbook.md,
            deployment.md, glossary.md, decisions/]
```

Six docs, codebase-aware, AI-retrievable. The architecture doc
points to the api-reference for endpoint detail; the runbook
references the deployment doc for rollback procedure; the glossary
defines domain terms once. AI agents pull only what they need per
task, fitting the context window.

For users who don't want the new behaviour: the prompt defaults to
`Y` for new bootstraps, but existing users on `v1.5.x → v1.6.0`
upgrade see no scaffold changes — `use_archetype_scaffolds` defaults
to `false` and `retrofit` does not flip it.

## Technical design

### Project Memory principles doc (already drafted)

[`docs/principles/documentation.md`](../principles/documentation.md)
ships as part of v1.6.0. Defines the term, the five properties, the
layered model (CLAUDE.md / Project Memory / `memory/` / Claude
auto-memory), the four-question test for new templates.

### `StackDescriptor.archetype` field

```json
{
  "archetype": "api-service" // one of seven values; "unknown" for fallback
}
```

Detection in `bin/detect-stack.sh`:

| Archetype | Signal precedence (first match wins) |
|---|---|
| `plugin` | `.claude-plugin/plugin.json`, `manifest.json` (browser ext), `extension.json` (VS Code) |
| `mobile-app` | profile is `swift-ios` / `kotlin-android` / `flutter-app` |
| `api-service` | `openapi.{yaml,yml,json}`, `*.proto`, `swagger.json`; OR (server framework detected AND no frontend framework) |
| `web-app` | frontend framework (`next.config.*`, `vite.config.*`, `nuxt.config.*`, `svelte.config.*`); OR (server framework AND frontend framework — full-stack) |
| `cli-tool` | `bin` field in `package.json`; `cmd/main.go` in Go; `console_scripts` in `setup.py`/`pyproject.toml`; profile is `*-cli` |
| `library` | published-package signals (no entry-point binary, profile marked `library`, `lib/` exports) |
| `unknown` | fallback |

Conflict resolution:
- Profile-declared `archetype` wins.
- Detection emits the suggestion as a hint when the profile leaves
  archetype unset.
- Detection mismatch with profile is a `warn`-level note in
  `doctor`, not a hard failure.

### Profile schema additions

Two new optional fields. Both additive — existing profiles validate
unchanged.

```json
{
  "archetype": "api-service",
  "documentation": {
    "use_archetype_scaffolds": false,
    "scaffold_types": ["architecture", "adrs", "research"]
  }
}
```

- `archetype`: one of the seven values; if absent, detection result
  is used.
- `documentation.use_archetype_scaffolds`: when `true`, `scaffold_types`
  is *replaced* by the per-archetype map (below). When `false` or
  absent, current flat-list behaviour is preserved.

### Per-archetype scaffold maps

Hard-coded in `bin/scaffold-docs.sh` (data-as-code; profile
overrides take precedence):

| Archetype | Scaffold types |
|---|---|
| `api-service` | architecture, api-reference, runbook, deployment, adrs, glossary |
| `cli-tool` | architecture, runbook, adrs, glossary |
| `library` | architecture, api-reference, adrs, glossary |
| `web-app` | architecture, runbook, deployment, adrs, glossary |
| `mobile-app` | architecture, runbook, deployment, adrs, glossary |
| `plugin` | architecture, adrs, glossary |
| `unknown` | architecture, adrs (matches today's default) |

`memory/` is excluded from the maps — it's the ephemeral scratch
layer, scaffolded unconditionally regardless of archetype.

### New templates (4 files)

All four go in `templates/docs/`. Each follows Project Memory
principles: bounded scope, predictable structure, codebase-aware
variables where possible.

#### `templates/docs/api-reference.tmpl`

For `api-service` and `library`. Structure:

```markdown
# API Reference

## Overview
> One paragraph: what this API does, base URL pattern, auth model.

## Conventions
> HTTP methods, error envelope, pagination, content types,
> rate limits.

## Endpoints
> One subsection per endpoint OR per resource group.
> Each: method, path, request shape, response shape, errors,
> auth requirements, idempotency notes.

## Versioning
> Compatibility model. How breaking changes ship.

## See also
- See `architecture.md` — how the API fits into the broader system.
- See `runbook.md` — operational procedures.
```

If `openapi.{yaml,json}` is detected at scaffold time, the template
includes a callout: *"This file should be kept in sync with
`openapi.yaml`. Consider auto-generating sections marked with
`<!-- generated -->`."*

#### `templates/docs/runbook.tmpl`

For `api-service`, `cli-tool`, `web-app`, `mobile-app`. Structure:

```markdown
# Runbook

## Operational invariants
> What MUST be true for this system to be healthy.

## Routine procedures
> Deploy, rollback, restart, scale up/down. One subsection each.

## Incident response
### Symptom: <observable problem>
- Likely causes
- Diagnostic commands
- Mitigation steps
- Rollback criteria

## Monitoring
> Where the dashboards live, what alerts fire, on-call rotation.

## See also
- See `deployment.md` — how the system ships.
- See `architecture.md` — system context.
```

The "Symptom: <observable problem>" pattern is deliberate — AI
agents debugging an outage retrieve by symptom, not by cause.

#### `templates/docs/deployment.tmpl`

For all archetypes except `library` and `plugin`. Structure:

```markdown
# Deployment

## Topology
> Where this runs in production. Hosts, regions, replicas.

## Pipeline
> How code reaches production. CI steps, gates, manual approvals.

## Configuration
> Env vars, secrets, feature flags, config files. Where each lives.

## Rollout model
> Blue/green, canary, rolling, immediate. Rollback procedure.

## Local-to-prod parity
> What's the same. What's different. Why.

## See also
- See `runbook.md` — operational procedures.
- See `architecture.md` — what's being deployed.
```

If a `Dockerfile`, `docker-compose.yml`, or `.github/workflows/`
is detected at scaffold time, the template fills in detected paths.

#### `templates/docs/glossary.tmpl`

For all archetypes (the sleeper hit). Structure:

```markdown
# Glossary

## Conventions
> One entry per term. Lead with a one-sentence definition. Follow
> with invariants, related terms, and where the term is used.

## Terms

### <Term>
**Definition.** One sentence.

**Invariants.** What MUST be true about this thing.

**Used in.** Files, modules, or docs where this term appears.

**Related.** Other glossary entries.

---

> Add new entries above. Keep entries short — link to other docs
> for detail.
```

Glossary entries are intentionally compact. AI agents retrieve
domain context one term at a time; bounded scope per entry matters
even within a single doc.

### `bin/scaffold-docs.sh` changes

```bash
# Pseudocode
if jq -e '.documentation.use_archetype_scaffolds' <profile> | grep -q true; then
  archetype=$(jq -r '.documentation.archetype // .stack.archetype // "unknown"' <plan>)
  scaffold_types=$(map_archetype_to_scaffolds "$archetype")
else
  scaffold_types=$(jq -r '.documentation.scaffold_types[]' <profile>)
fi

for t in $scaffold_types; do
  scaffold_template "$t"
done
```

Backward compat: when `use_archetype_scaffolds` is `false` (default),
the existing flat-list path runs unchanged. Existing users see no
behaviour change.

### Drift-check dedup

#### `bin/session-check.sh` change

Add `--flow=<commit|release|pr|ship>` argument. When set, the script
emits the existing one-line drift summary *plus* the framed nudge:

```text
Heads up: nyann detected drift vs your profile. Run /nyann:retrofit
when you get a chance. (Non-blocking — proceeding with the <flow>
flow.)
```

Verb mapping: `commit` → `commit`, `release` → `release`, `pr` →
`PR`, `ship` → `ship`.

When `--flow` is unset, current behaviour is preserved (used by other
callers like CI). New flag is opt-in for the four caller skills.

#### Trim 4 SKILL.md preambles

In `commit/SKILL.md`, `release/SKILL.md`, `pr/SKILL.md`, and
`ship/SKILL.md`, replace the 8-line preamble with:

```markdown
## 0. Drift check (quick, non-blocking)

Run `bin/session-check.sh --flow=<commit|release|pr|ship>` (use the
flag matching this skill). Surface its output verbatim if any. Do
not block on it.
```

Saves ~370 B per skill body × 4 = ~1.5 KB. Future wording changes
need one edit, not four.

### README positioning update

Add a new section after `What you get`:

```markdown
## Project Memory

nyann scaffolds and maintains your project's **Project Memory** — a
documentation layer designed for AI agents to retrieve, sized to fit
their context, and kept in sync as your code evolves.

Project Memory has five properties:

1. AI-retrieval-first — bounded scope per doc, predictable structure
2. Size-budgeted — fits in context windows, no overload
3. Drift-aware — broken links, orphans, staleness surfaced automatically
4. Storage-agnostic — local, Obsidian (MCP), or Notion (MCP)
5. Dual-audience — works for AI agents AND humans

See `docs/principles/documentation.md` for the full definition.
```

## Testing strategy

### New bats

- **`test-detect-archetype.bats`** — fixture repos for each of the
  seven archetypes; assert correct detection. Negative cases:
  multi-stack ambiguity (frontend + backend → `web-app` per
  precedence), profile override (profile says `cli-tool` but
  signals say `api-service` → profile wins, hint emitted).
- **`test-archetype-scaffolds.bats`** — for each archetype, run
  `scaffold-docs.sh` with `use_archetype_scaffolds: true` and assert
  the per-archetype map produces the expected file set. Verify
  `false`/absent flag preserves flat-list behaviour.
- **`test-templates-archetype.bats`** — render each new template
  against fixture stack descriptors. Assert codebase-aware variables
  populate correctly. Assert idempotency (re-run does not
  overwrite user content).
- **`test-session-check-flow.bats`** — `--flow=<commit|release|pr|ship>`
  emits the correct verb in the nudge. Unset `--flow` preserves
  pre-v1.6.0 output for other callers.
- **`test-claudemd-self-compliance.bats`** — assert this repo's own
  `CLAUDE.md` size ≤ 3 KB soft cap. Guards against future regression.
- **`test-memory-readme-content.bats`** — assert
  `templates/memory/README.tmpl` mentions Claude Code's auto-memory
  location (the framing fix's load-bearing line).

### Existing bats updates

- **`test-detect-stack.bats`** — assert `archetype` field present
  in StackDescriptor output for each fixture.
- **`test-bootstrap.bats`** — assert opt-in flag flow works end-to-end.
- **`test-retrofit.bats`** — assert opt-in flag does not auto-flip
  on retrofit.
- **`test-doctor.bats`** — assert archetype-mismatch surfaces as a
  `warn`-level note when profile and detection disagree.

No new evals are blocking for v1.6.0. The 8 skills currently

`gen-claudemd`, `gh-protect`, `hotfix`, `route-docs`, `ship`,

## Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| Archetype detection misclassifies edge cases | Low | Opt-in flag; profile override; mismatch surfaces as `warn` not error |
| New templates render imperfectly first time | Low — opt-in flag keeps blast radius small | Soft launch via opt-in; iterate in v1.6.x patches |
| Memory framing fix confuses existing users | Low — folder structure unchanged | Migration note in CHANGELOG; framing is text-only |
| CLAUDE.md self-fix breaks the repo's own contributor flow | Low — only this repo | PR review + bats `test-claudemd-self-compliance` |
| Drift-check `--flow` arg breaks non-skill callers of `session-check.sh` | Zero | Flag is additive; default behaviour preserved |
| Profile schema additions break existing profiles | Zero | Both new fields optional |
| Multi-archetype monorepos detected as a single archetype | Medium — real for some users | Documented limitation; profile override available; multi-archetype is a v1.7+ design problem |
| Glossary template feels "too empty" without auto-population | Medium | Header + one filled-in example term per archetype to seed the pattern |
| Notion/Obsidian users get the same Markdown templates | Low — works fine, just not native-formatted | Acceptable for v1.6.0; native variants are a v1.7+ question |

## Open questions

1. **Should Phase 1's CLAUDE.md self-fix fold "Non-negotiable
   conventions" into `docs/principles/conventions.md`, or is that
   a separate principles doc?** Current draft has them as separate
   files. Keeping them separate makes each doc bounded-scope; folding
   them risks a "principles dumping ground." Lean toward separate.

2. **For `library` archetype, should `api-reference.tmpl` be
   identical to the api-service variant?** Probably similar but
   not identical — libraries document programmatic APIs (functions,
   classes, traits), API services document HTTP. May warrant a second
   template (`library-api-reference.tmpl`) in v1.7+. For v1.6.0, ship
   one template with internal sub-sections for each style.

3. **Should `bootstrap-project` default the opt-in prompt to `Y`
   or `N`?** Lean `Y` — archetype-aware is the recommended path
   forward. But `Y` defaults could surprise users if detection is
   wrong on their first run. Consider `Y` for v1.6.0, watch for
   complaints, flip to `N` only if wrong-detection is common.

4. **Does retrofit ever offer to flip `use_archetype_scaffolds` from
   `false` to `true`?** Per-decision (D3): no, opt-in stays opt-in.
   But should retrofit at least *show* what the new scaffolds would
   be if flipped, as informational? Lean yes — surface as
   `informational` drift, not actionable.

5. **Glossary auto-seeding from existing CLAUDE.md / README terms.**
   Could v1.6.0 do a one-time scan of existing docs for capitalized
   noun phrases and pre-populate glossary entries? Probably noisy;
   defer to v1.7+ when type-graph extraction makes more sense.
