# v1.5.0 — Release Automation

**Status:** Draft (proposed, not yet implemented)
**Owner:** project maintainer
**Last updated:** 2026-05-03

## Problem

Cutting a release for nyann today requires the maintainer to do six things by
hand. `bin/release.sh` covers two of them; the rest are off the wedge and easy
to forget.

| Step | Today | What handles it |
|------|-------|-----------------|
| 1. Bump `.claude-plugin/plugin.json` version | manual edit | nothing |
| 2. Bump `.claude-plugin/marketplace.json` version | manual edit | nothing |
| 3. Write `CHANGELOG.md` entry for the release | manual OR `bin/release.sh` | `release.sh --strategy conventional-changelog` |
| 4. Update `README.md` "Shipped in vX.Y.Z" section | manual edit | nothing |
| 5. Create + push annotated git tag | `bin/release.sh --push` | `release.sh` |
| 6. Create GitHub release with notes | manual `gh release create` | nothing |

Empirical evidence the gap is real: `marketplace.json` was stale at `1.2.1`
through the v1.3.0 release and was caught only during the v1.4.0 cut by manual
inspection. The current pattern requires the maintainer to remember every
file and step, which doesn't scale and is exactly the kind of failure mode
nyann exists to prevent.

## Goals (P0)

- One command — `/nyann:release 1.5.0` — drives the full cut from start to
  finish.
- Manifest version bumps are profile-declared. Each stack's profile lists
  which files carry the version string and how to mutate them. nyann itself
  dogfoods this for `plugin.json` + `marketplace.json`.
- GitHub release creation is automatic after the tag push, with notes
  auto-extracted from the freshly-prepended `CHANGELOG.md` section.
- Backward compatible — every existing `bin/release.sh` invocation keeps
  working unchanged. New flags are opt-in.

## Non-goals (explicit)

- **`README.md` "Shipped in" section updates.** This is nyann-specific
  structure; other repos won't have it. We will document the pattern in
  `CLAUDE.md` so future maintainers remember to update it manually, but
  automating it requires a generic markdown-block primitive that's a
  bigger design problem and not justified by N=1.
- **Auto-publishing to npm/PyPI/cargo/Hex/etc.** Out of scope — those are
  package-manager concerns with their own auth and policy. nyann stops at
  the git tag + GitHub release.
- **Calendar versioning, build numbers, or non-semver schemes.** Today's
  `release.sh` already enforces semver; that constraint stays.
- **Auto-detection of which files carry version strings.** Heuristics
  (look at every package.json, look for "version" keys) sound nice but
  silently break. Profiles list files explicitly.

## User flow

### Before (today)

```bash
$ vim .claude-plugin/plugin.json       # bump version
$ vim .claude-plugin/marketplace.json  # bump version (forgot? oh well)
$ vim CHANGELOG.md                     # write entry by hand
$ vim README.md                        # add "Shipped in vX.Y.Z" section
$ git commit -am "release: v1.5.0 prep"
$ bin/release.sh --version 1.5.0 --push --yes
$ awk '/^## \[1\.5\.0\]/,/^## \[1\.4\.0\]/' CHANGELOG.md \
    | sed '$d' > /tmp/notes.md
$ gh release create v1.5.0 --notes-file /tmp/notes.md
```

Six commands, three of which are manual edits. Forgetting any one of the
edits is silent — `marketplace.json` proved this.

### After (proposed)

```bash
$ /nyann:release 1.5.0
[shows preview: bumps + CHANGELOG block + tag + GH release]
[user confirms]
[done]
```

One command. The preview-before-mutate flow already in `release.sh`
(`--yes` confirmation) extends to cover the manifest bumps + GH release.

## Technical design

### Profile schema additions

New optional `release` block. Today's profiles have no `release` section;
introducing one is additive and requires no migration.

```json
{
  "release": {
    "bump_files": [
      {
        "path": ".claude-plugin/plugin.json",
        "format": "json-version-key",
        "key": ".version"
      },
      {
        "path": ".claude-plugin/marketplace.json",
        "format": "json-version-key",
        "key": ".plugins[0].version"
      }
    ]
  }
}
```

Supported `format` values for v1.5.0:

- `json-version-key` — uses `jq` to set the named key. Implementation:
  `jq --arg v "$VERSION" '<key> = $v' file > file.tmp && mv file.tmp file`.
  Covers `plugin.json`, `marketplace.json`, `package.json`, and any JSON
  manifest with a settable version path.
- `toml-version-key` — sed/awk on a single-line `version = "x.y.z"` inside
  a named TOML section. Targets `pyproject.toml` (`[project]`) and
  `Cargo.toml` (`[package]`). Multi-line strings, inline tables, and
  nested sections die loudly with a clear "this script can't parse your
  TOML; bump it manually" error. Defensible scope for v1.5.0.
- `script` — escape hatch. Profile says `{format: "script", command: "..."}`
  and the script gets `$NEW_VERSION` in env. For Hex (`mix.exs`), Gleam,
  and anything we haven't built explicit support for. User-owned, no
  warranty.

### New `bin/release.sh` flags

#### `--bump-manifests`

Before the CHANGELOG prepend (so the bumps land in the same release commit),
read `release.bump_files[]` from the resolved profile. For each entry, edit
the file in place to set the new version. Track the modified files for
inclusion in `git add`.

Failure modes:
- Profile has no `release.bump_files` → log `"no bump_files configured;
  skipping --bump-manifests"` and proceed.
- Manifest file doesn't exist → die with file path.
- jq/sed mutation fails → die with the file path + format.
- File version key already matches `$VERSION` → log `"already at $VERSION,
  no change"` and continue (idempotent).

Combination rules:
- `--bump-manifests` + `--strategy manual` → die. Manual strategy doesn't
  create a commit, so there's no place for the bumps to land.
- `--bump-manifests` + `--dry-run` → preview the diff that would be applied,
  don't mutate.

#### `--gh-release`

After the tag push succeeds, invoke:

```bash
gh release create "$tag" \
  --title "${tag}" \
  --notes-file <tmp-with-rendered-changelog-block>
```

The notes file is the same rendered CHANGELOG block that just got prepended.
For prereleases (`-rc.N`, `-beta.N`), pass `--prerelease`.

Failure modes (matching nyann's gh integration convention):
- gh missing or unauthed → soft-skip with `gh_release.outcome = "skipped"`,
  `next_steps[]` includes the manual `gh release create` invocation. Tag
  + push are NOT undone.
- `gh release create` rc≠0 (e.g. release already exists, network error) →
  populate `gh_release.error`, surface in `next_steps[]`. Tag + push are
  NOT undone.

Combination rules:
- `--gh-release` without `--push` → die. Tag has to be on origin for a
  GitHub release to attach to it.
- `--gh-release` + `--strategy manual` → still works. Manual strategy
  creates the tag without CHANGELOG; the GH release uses `--generate-notes`
  in that case (gh's auto-generated commit list).

### `bin/release.sh` execution order

```
1. validate args (incl. --bump-manifests + manual conflict)
2. gh auth check (if --gh-release)
3. collect commits in range
4. render CHANGELOG block
5. if --bump-manifests:
     load profile.release.bump_files[]
     for each entry: apply bump, track file
6. CI gate (if --wait-for-checks)
7. if conventional-changelog AND not prerelease:
     prepend CHANGELOG.md
     git add CHANGELOG.md + bumped_files
     git commit -m "chore(release): vX.Y.Z"
8. git tag -a vX.Y.Z
9. if --push: push tag, push branch
10. if --gh-release: gh release create
11. emit ReleaseResult JSON
```

### Schema updates

- **`profiles/_schema.json`**: add `release` object with `bump_files[]`
  array. All fields optional. Each entry is a discriminated union on
  `format`.
- **`schemas/release-result.schema.json`**: add `bumped_files[]` (list of
  paths actually mutated) and `gh_release` object
  (`{outcome, url?, error?}`). Both optional to preserve backward compat.

### Skill layer updates

- **`skills/release/SKILL.md`**: when the active profile has
  `release.bump_files`, default the underlying invocation to
  `--bump-manifests --gh-release`. User can opt out via
  `bypass_bump_manifests: true` in the slash-command args.
- The skill's preview-step renders the bumps + CHANGELOG diff + GH release
  intent in one block before the `--yes` confirmation.

### nyann's own dogfood (Phase 3)

- Update `profiles/default.json` to declare:
  ```json
  "release": {
    "bump_files": [
      { "path": ".claude-plugin/plugin.json", "format": "json-version-key", "key": ".version" },
      { "path": ".claude-plugin/marketplace.json", "format": "json-version-key", "key": ".plugins[0].version" }
    ]
  }
  ```
- Update relevant starter profiles (`nextjs-prototype`, `python-cli`, etc.)
  to suggest the conventional manifest path for their stack.

## Testing strategy

New bats:

- **`test-release-bump-manifests.bats`** — verify `--bump-manifests` correctly
  mutates JSON files via the configured `key`, leaves untouched files
  alone, dies on missing files / invalid keys / unsupported formats, and
  is idempotent (re-run on already-bumped file is a no-op).
- **`test-release-gh-release.bats`** — mock `gh` (existing fixture pattern)
  to verify the GH release is created with the right title, notes file,
  and `--prerelease` flag for `-rc.N` versions. Soft-skip path verified.
- **`test-release-bump-toml.bats`** — covers the TOML bump path against
  fixture `pyproject.toml` and `Cargo.toml` files. Negative cases:
  multiline strings, inline tables, missing section header.

Extensions to existing tests:

- **`test-release.bats`** — verify `--bump-manifests --strategy manual`
  dies with a clear error.
- **`test-release.bats`** — verify dry-run shows the bump diff without
  mutating.

Schema:

- **`test-profile-schema.bats`** — verify a profile with `release.bump_files`
  validates; verify required-field enforcement on each entry.

## Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| TOML editing without a real parser | Medium — pyproject.toml + Cargo.toml are real targets | Scope to single-line `version = "x.y.z"` within a known section; die loudly on anything else. Future ADR can move to `dasel` or `tomlq` if the fragility bites. |
| Tag pushed but GH release fails (network race) | Low — already handled by `next_steps[]` pattern | Surface the failure in `next_steps[]`; user re-runs `gh release create` manually. Don't try to undo the tag — that breaks the marketplace contract. |
| User runs `--bump-manifests` against a profile without `release.bump_files` | Low — log and skip | Already in the design (no-op with log line). |
| Profile schema migration breaks existing profiles | Zero — additive | `release` is optional; profiles without it are valid. |
| GH-release notes contain markdown that breaks `gh` CLI parsing | Low — uses `--notes-file` which is byte-exact | `gh release create --notes-file` reads bytes, no shell interpolation in the notes content. |

## Phasing

Three phases, each shippable independently. If Phase 2 or 3 slip, Phase 1
still shipped value.

### Phase 1 — Manifest bumps (P0)
- `--bump-manifests` flag on `bin/release.sh`
- `json-version-key` + `toml-version-key` + `script` formats
- Profile schema update (`release.bump_files[]`)
- Bats coverage
- Update `RELEASING.md` to mention the new flag

**Smallest delta.** Removes the `marketplace.json`-staleness class of bugs
forever.

### Phase 2 — GitHub release creation (P0)
- `--gh-release` flag on `bin/release.sh`
- Soft-skip + `next_steps[]` integration
- Bats coverage with mocked `gh`

**Independent of Phase 1.** Could ship first if simpler.

### Phase 3 — Skill + dogfood (P1)
- `skills/release/SKILL.md` defaults to `--bump-manifests --gh-release`
  when profile has bump_files
- `profiles/default.json` declares nyann's own bump_files
- Starter profiles updated with stack-typical manifest paths
  (`package.json`, `pyproject.toml`, `Cargo.toml`)
- README "Shipped in v1.5.0" section (manual, per non-goals decision)

**Locks in the win.** After this, `/nyann:release 1.5.0` for nyann itself
is the one-command flow.

## Open questions

1. **Should `--bump-manifests` be the default rather than opt-in?**
   Argument for: profile declares the files; if it didn't want bumps, it
   wouldn't list them. Argument against: existing `release.sh` callers
   don't expect side effects on files they didn't pass via flag. Lean
   toward opt-in for v1.5.0; consider flipping the default in v2.0 with a
   migration note.

2. **TOML format scope** — pin to single-line `version = "x.y.z"` only, or
   try harder (multi-line tables, inline `version` in `[tool.poetry]`)?
   Lean toward minimal scope: keep us out of the TOML-parsing rabbit hole
   for v1.5.0, document the constraint, expand if a real profile needs it.

3. **Per-file commit message override** — should each `bump_files[]` entry
   support a custom commit message scope (e.g. `chore(plugin): bump to
   1.5.0`)? Probably not; one consolidated `chore(release): vX.Y.Z` is
   cleaner. Skip for v1.5.0.

4. **Should `--gh-release` populate the release with assets** (compiled
   binaries, plugin tarballs)? Out of scope for v1.5.0 — assets are stack-
   specific. Maybe a `release.assets[]` profile field in v1.6.0+.

## Acceptance criteria

For Phase 1+2 together to ship as v1.5.0:

- [ ] `bin/release.sh --bump-manifests --version X.Y.Z` mutates every file
      listed in `release.bump_files[]` of the active profile, in the same
      release commit as CHANGELOG.md.
- [ ] `bin/release.sh --gh-release --push --version X.Y.Z` creates a
      GitHub release attached to the pushed tag with the rendered CHANGELOG
      block as notes.
- [ ] Both flags soft-fail gracefully (skipped reason in JSON, non-zero
      next_steps[]) when their preconditions aren't met.
- [ ] All new bats pass; existing bats unchanged.
- [ ] `/nyann:release X.Y.Z` against a profile with `release.bump_files`
      drives the full flow including preview confirmation.
- [ ] `RELEASING.md` updated to reflect the new flow.
- [ ] nyann's own `default` profile carries the bump_files entries for
      `plugin.json` + `marketplace.json` (Phase 3).
